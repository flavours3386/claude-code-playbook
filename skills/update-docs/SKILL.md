# update-docs

프로젝트 내 모든 .md 문서를 스크리닝하고, 현재 코드/설정 상태와 비교하여 불일치하는 부분을 일괄 업데이트한다.

## 왜 이 스킬이 필요한가

코드를 변경하면 관련 문서(CLAUDE.md, ARCHITECTURE.md 등)도 함께 업데이트해야 하지만, 어떤 문서가 있는지 모르거나 잊어서 누락되는 경우가 많다. 이 스킬은 프로젝트 내 모든 .md를 자동으로 찾아 코드와 비교함으로써 문서 누락을 방지한다.

## 워크플로우

### 1단계: 프로젝트 문서 스캔

프로젝트 루트에서 모든 .md 파일을 수집한다.

```
glob **/*.md
```

아래 디렉토리는 반드시 제외한다 (라이브러리/빌드 산출물이므로):
- `node_modules/`
- `.venv/`, `venv/`
- `.git/`
- `dist/`, `build/`
- `.next/`
- `__pycache__/`
- `.claude/`

대상은 프로젝트가 직접 관리하는 문서만이다. 보통 루트와 `docs/` 디렉토리에 위치한다.

### 2단계: 코드 상태 수집

문서와 비교할 "현재 코드의 진실"을 수집한다. git diff가 아닌 **코드 직접 읽기**가 주 전략이다.

**소스 파일 식별 우선순위:**
1. 진입점 파일 (main.py, index.ts, Code.js, app.py 등)
2. 설정 파일 (package.json, pyproject.toml, appsscript.json, .env.example 등)
3. 라우트/API 정의 파일
4. 최근 변경된 소스 파일 (git log --name-only -5 참고)

대규모 프로젝트(소스 파일 20개 이상)에서는 진입점 + 설정 + 최근 변경 파일을 우선 읽고, 문서에서 언급된 특정 파일만 추가로 확인한다.

**추출 대상:**
- 함수/클래스 목록 (특히 public API, 메뉴, CLI 명령어)
- 설정 상수, 환경변수
- 기술 스택, 의존성
- 폴더 구조
- 트리거, 스케줄, 배포 설정

**git log는 보조 수단:**
- `git log --oneline -10`으로 최근 커밋 이력 확인
- CLAUDE.md 등의 "최근 변경사항" 섹션 갱신 용도로만 사용

### 3단계: 문서별 비교 + 업데이트

각 .md 파일을 읽고, 코드에서 수집한 정보와 비교하여 불일치 항목을 수정한다.

**문서 역할은 파일명과 내용으로 동적 판별한다.** project-setup 표준 구조가 아닌 프로젝트에서도 동작해야 하므로, 아래는 참고용이지 하드코딩하지 않는다:

| 파일명 패턴 | 일반적 역할 | 주요 검증 항목 |
|------------|-----------|-------------|
| CLAUDE.md | 프로젝트 개요, 진입 계약서 | 기술 스택, 명령어, 메뉴 구조, 폴더 구조, 최근 변경사항 |
| README.md | 프로젝트 소개 | 설치 방법, 사용법, 기술 스택 |
| ARCHITECTURE.md | 아키텍처 | 레이어 구조, 데이터 흐름, 외부 의존성 |
| AGENTS.md | 목차, 핵심 원칙 | Quick Start 절차, 핵심 원칙 |
| PLANS.md | 로드맵, 기술 부채 | 완료 체크리스트, 기술 부채 목록 |
| PRODUCT_SENSE.md | 제품 방향 | 성공 지표 (큰 변경 시에만) |
| CHANGELOG.md | 변경 이력 | git log 기반 갱신 |
| API.md | API 명세 | 엔드포인트, 파라미터 |
| 기타 .md | 내용에 따라 판단 | 코드에서 검증 가능한 기술 사실만 |

**업데이트 순서** (의존 관계를 반영하여 상위 문서부터):
1. 아키텍처 문서 (ARCHITECTURE.md 등 - 기술 사실의 기반)
2. 진입 문서 (CLAUDE.md, README.md 등 - 아키텍처 참조)
3. 가이드 문서 (AGENTS.md 등)
4. 계획 문서 (PLANS.md 등)
5. 제품 문서 (PRODUCT_SENSE.md 등 - 큰 변경 시에만)
6. 기타 문서

**3단계 진실 계층 (이것이 핵심이다):**

- **Tier 1 - 코드/설정이 권위**: 함수명, 상수값, 파일 구조, API 엔드포인트, 메뉴 항목, 트리거 설정 등 코드에서 직접 확인 가능한 사실. 문서와 코드가 다르면 **코드 기준으로 문서를 수정**한다.

- **Tier 2 - git log가 권위**: 변경 이력, 날짜, 커밋 내용. "최근 변경사항" 같은 시간순 섹션은 **git log 기반으로 갱신**한다.

- **Tier 3 - 기존 문서 보존**: 설계 의도, 비즈니스 맥락, "왜 X를 선택했는가" 같은 내용은 코드에서 검증할 수 없다. **수정하지 않고 기존 내용을 보존**한다. 의심스러운 불일치가 있으면 결과 보고에서 플래그만 남긴다.

이 계층이 중요한 이유: 코드에서 확인할 수 없는 내용을 추측으로 수정하면 오히려 정확한 정보를 망가뜨릴 수 있다. 확실한 것만 고치고, 불확실한 것은 건드리지 않는 것이 안전하다.

### 4단계: 워크스페이스 문서 연동

프로젝트가 워크스페이스(멀티 프로젝트 루트) 하위에 있는지 감지하고, 워크스페이스 레벨 문서도 함께 업데이트한다.

**워크스페이스 감지 방법:**
1. 현재 프로젝트 디렉토리의 상위 경로를 탐색한다 (최대 2단계: `../`, `../../`)
2. 상위 디렉토리에 CLAUDE.md가 있고, 그 안에 프로젝트 테이블(마크다운 테이블 `| 프로젝트 | 용도 | 스택 |` 패턴)이 포함되어 있으면 워크스페이스로 판단한다

**감지 실패 시:** 워크스페이스가 없는 독립 프로젝트로 간주하고 이 단계를 스킵한다.

**워크스페이스 문서 업데이트 대상:**

| 문서 | 검증 항목 | 업데이트 방법 |
|------|----------|-------------|
| CLAUDE.md | 프로젝트 테이블에 현재 프로젝트가 등록되어 있는가 | 없으면 해당 섹션(Work 또는 Personal)의 테이블에 행 추가. 있으면 용도/스택이 프로젝트 CLAUDE.md와 일치하는지 확인하고 불일치 시 갱신 |
| ARCHITECTURE.md | 교차 영향 섹션에 현재 프로젝트 관련 내용이 있는가 | 새 프로젝트가 기존 연관 그룹에 영향을 주는 경우에만 플래그. 자동 수정하지 않음 (Tier 3) |

**워크스페이스 업데이트 규칙:**
- 프로젝트 CLAUDE.md에서 프로젝트명, 한줄 설명, 기술 스택을 추출한다
- 워크스페이스 CLAUDE.md의 테이블 형식(`| 프로젝트 | 용도 | 스택 |`)에 맞춰 행을 추가/갱신한다
- 연관 그룹 판단이나 교차 영향 분석은 자동으로 하지 않는다 (Tier 3 — 수동 확인 필요로 플래그)
- 워크스페이스 문서의 다른 섹션(설계 원칙, Design Context 등)은 건드리지 않는다

### 5단계: 결과 보고

업데이트 완료 후 아래 형식으로 보고한다:

```
## 문서 업데이트 결과

### 업데이트됨
- CLAUDE.md: 메뉴 구조 테이블 갱신 (2개 항목 추가), 트리거 주기 15분->5분 수정
- ARCHITECTURE.md: 데이터 흐름 다이어그램 업데이트

### 워크스페이스 문서
- ../../CLAUDE.md: 독립 프로젝트 테이블에 my-project 행 추가

### 스킵 (변경 불필요)
- AGENTS.md: 코드와 일치, 변경 없음
- PLANS.md: 코드와 일치, 변경 없음

### 수동 확인 필요
- PRODUCT_SENSE.md: 성공 지표 "매칭률 94%"가 현재 96.8%일 수 있음 (코드에서 확인 불가, Tier 3)
- ../../ARCHITECTURE.md: 새 프로젝트가 기존 연관 그룹에 영향을 주는지 수동 확인 필요
```

## 주의사항

- **변경 없으면 건드리지 않는다.** 이미 최신 상태인 문서를 불필요하게 수정하지 않는다.
- **추측하지 않는다.** 코드에서 확인할 수 없는 내용은 보존한다. "아마 이렇게 바뀌었을 것"으로 문서를 수정하면 안 된다.
- **비표준 문서도 처리한다.** CLAUDE.md, ARCHITECTURE.md 같은 표준 문서가 없는 프로젝트에서도 동작한다. README.md, CHANGELOG.md 등 어떤 .md든 코드 상태와 비교하여 업데이트한다.
- **워크스페이스 문서는 테이블 행 추가/갱신만 한다.** 설계 원칙, 연관 그룹, 교차 영향 등 구조적 변경은 자동으로 하지 않고 플래그만 남긴다.
